# 2026-06-02 멘토링 질문

## 목적

Terraform으로 AWS 인프라를 구성할 때 로컬 개발 환경, 저비용 AWS 검증 환경, 서비스별 배포 전략, 운영 배포 후 문제 판단 기준, 최근 현업의 AI 활용 방식, 프로젝트의 포트폴리오 방향성을 어떻게 바라보면 좋은지 질문한다.

## 질문 목록

1. 한 줄 질문: 현업에서는 로컬 Kubernetes 환경과 AWS 검증 환경을 어떻게 나눠서 구성하나요?

   상세 질문: Terraform으로 AWS 구성을 만들 때 개발자들이 개인적으로 쓰는 로컬 개발 환경은 보통 어떻게 구성하는가? 초기에는 Vagrant와 VMware 안에서 Kubernetes 클러스터를 구성했지만, 노드 실행과 환경 구성에 시간이 많이 걸려서 로컬은 Docker Desktop Kubernetes를 쓰기로 했다. 이 구성이 AWS에서 실제로 잘 돌아가는지는 현업에서 주로 어떤 방식으로 검증하는지 궁금하다.

2. 한 줄 질문: 제한된 AWS 예산에서는 Kubernetes 노드 구성을 어떤 기준으로 잡나요?

   상세 질문: AWS 예산이 월 20만원으로 한정된 상황에서 Kubernetes 노드 구성을 어떻게 잡는 것이 현실적인가? 현재는 메모리 부족을 줄이기 위해 `r6g` 인스턴스 3대로 실행하고 있는데, 현업에서는 비용과 안정성 사이에서 보통 어떤 구조를 선택하는지 궁금하다.

3. 한 줄 질문: 서비스마다 배포 전략은 어떤 기준으로 다르게 정하나요?

   상세 질문: 서비스가 6개 있을 때 각 서비스마다 적합한 배포 전략을 어떻게 정하는지 궁금하다. 단순히 롤링 업데이트, 블루/그린, 카나리 중 무엇을 쓰는지가 아니라, 결제, 예약, 좌석 점유, 알림처럼 서비스마다 장애 영향, 데이터 정합성, 롤백 난이도가 다를 때 현업에서는 어떤 기준으로 배포 방식을 나누는지 알고 싶다. 특히 DB 마이그레이션, 이벤트 메시지, 외부 API 연동처럼 단순 롤백이 어려운 경우에는 어떤 방식으로 대응하는지도 궁금하다.

4. 한 줄 질문: 현업에서는 AI를 개발과 운영 업무에 실제로 어떻게 활용하고 있나요?

   상세 질문: 최근 현업에서는 AI를 이미 개발과 운영 과정에 쓰고 있을 텐데, 실제로 어떤 방식으로 활용 중인지 궁금하다. 코드 작성 보조뿐 아니라 문서 정리, 테스트 작성, 장애 분석, 로그 요약, 코드 리뷰, 배포 자동화 같은 업무에서 어느 정도까지 쓰고 있는지 알고 싶다. 팀 안에서 AI를 사용할 때 정해둔 기준이나 조심하는 부분이 있는지도 궁금하다.

5. 한 줄 질문: 포트폴리오 관점에서 이 프로젝트는 무엇에 집중해서 발전시키는 것이 좋을까요?

   상세 질문: 포트폴리오 관점에서 이 프로젝트는 앞으로 무엇에 집중하고 어떤 방향으로 발전시키는 것이 좋은지 궁금하다. 단순히 Kubernetes와 AWS 인프라를 구성했다는 점을 보여주는 것에서 더 나아가, 실제 서비스 관점에서 어떤 문제를 풀고 어떤 역량을 드러내야 하는지 알고 싶다. 최근에는 프로젝트에 AI 서비스를 하나 붙여보거나 AI를 활용한 기능을 보여주는 경우도 있다고 들었는데, 이 프로젝트에서도 그런 방향을 고려하는 것이 좋은지 궁금하다.

6. 한 줄 질문: 운영 배포 후에는 어떤 지표와 기준으로 문제가 생겼다고 판단하나요?

   상세 질문: 개발, 검증, 사인오프까지 끝난 뒤 운영 배포를 하고 나면, 사후 모니터링에서 무엇을 근거로 문제가 생겼다고 판단하는지 궁금하다. 단순히 Pod 상태나 에러 로그만 보는 것이 아니라, 에러율, 응답 시간, 큐 적체, DB 상태, 결제/예약 성공률 같은 기술 지표와 비즈니스 지표를 어떻게 함께 보고 배포 성공, 관찰 지속, 롤백 여부를 판단하는지 알고 싶다.

## 현재 전제

| 항목 | 현재 방향 |
| --- | --- |
| 로컬 Kubernetes | Docker Desktop Kubernetes 사용 |
| 이전 로컬 구성 | Vagrant + VMware 내부 Kubernetes 클러스터 |
| AWS 노드 | `r6g` 계열 3대 |
| 서비스 구성 | 6개 서비스 |
| 배포 전략 후보 | 롤링 업데이트, 블루/그린, 카나리 |
| 배포 전략 판단 기준 | 장애 영향, 데이터 정합성, 롤백 난이도, 외부 연동 |
| 운영 배포 판단 기준 | 에러율, 응답 시간, 큐 적체, DB 상태, 결제/예약 성공률 |
| AI 활용 관심사 | 현업 활용 사례, 팀 사용 기준, 보안과 품질 주의점 |
| 포트폴리오 방향성 | 집중할 역량, 실제 서비스 관점, AI 기능 또는 AI 활용 가능성 |
| 주요 제약 | AWS 예산 제한, 메모리 부족 회피, 로컬 구성 시간 단축 |

## 후속 정리 필요

- 답변을 받은 뒤 로컬 개발 환경 기준은 ADR 또는 온보딩 문서에 반영한다.
- AWS 비용과 노드 구성 기준은 `docs/projects_plan/reference/AWS_INFRA_SIZING.md`에 반영할지 확인한다.
- 서비스별 배포 전략 기준은 카나리 배포/롤백 시나리오 또는 GitOps 작업 범위에 반영할지 확인한다.
- 운영 배포 후 모니터링 기준은 관측성 지표 문서나 배포/롤백 시나리오에 반영할지 확인한다.
- AI 활용 사례는 팀 작업 방식이나 개인 개발 환경 정리 문서로 따로 남길지 확인한다.
- 포트폴리오 방향성 답변은 프로젝트 목표나 이후 작업 우선순위에 반영할지 확인한다.
