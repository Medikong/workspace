# 라이브 이벤트 커머스 운영 검증

## 문서 목적

이 문서는 클라우드 네이티브 과정의 심화 프로젝트 주제로 `라이브 이벤트 커머스 플랫폼`을 제안한다. 최종 목표는 완성형 커머스 서비스를 만드는 것이 아니라, 라이브 이벤트 중 발생하는 부하와 장애를 Kubernetes, 모니터링, 관측성, 배포 관점에서 검증하고 발표 가능한 증거를 확보하는 것이다.

## 프로젝트 한 줄 정의

라이브 방송 중 한정 상품 주문이 폭주해도, 주문·결제·채팅 장애를 격리하고 관측 가능한 방식으로 안정적으로 판매를 처리하는 클라우드 네이티브 커머스 운영 검증 프로젝트다.

## 핵심 검증 질문

> 라이브 이벤트 중 트래픽이 급증해도 핵심 구매 흐름을 유지하고, 장애를 격리하며, 운영자가 지표와 로그로 상태를 판단할 수 있는가?

## 왜 이 주제인가

라이브 커머스는 실시간성과 커머스 흐름이 결합되어 장애 조건이 자연스럽게 발생한다.

- 방송 시작 직후 접속자가 급증한다.
- 한정 상품 공개 순간 주문 API가 폭주한다.
- 채팅 메시지 증가로 로그와 이벤트 처리량이 함께 증가한다.
- 재고 차감 경쟁 조건이 발생할 수 있다.
- 결제 승인 지연이나 실패가 주문 흐름에 영향을 준다.
- 알림 지연 또는 실패가 사용자 경험에 영향을 준다.
- 배포 직후 에러율 상승이 매출 손실로 연결될 수 있다.

따라서 이 주제는 부하 테스트, 장애 격리, 관측성, 무중단 배포, 보안 정책을 억지 없이 하나의 서비스 시나리오에 연결할 수 있다.

## 주제 선정 근거

1. 클라우드 네이티브 운영 검증에 적합하다.

   - 라이브 이벤트는 특정 시점에 트래픽이 몰리는 구조라 부하 테스트와 확장성 검증을 설명하기 쉽다.
   - 상품 오픈, 주문, 결제, 채팅이 동시에 발생하므로 여러 서비스의 배포와 운영 상태를 함께 볼 수 있다.
   - 앱 기능보다 Kubernetes 배포, HPA, 장애 격리, 관측성 같은 운영 요소를 중심에 두기 좋다.

2. 정량 지표를 만들기 쉽다.

   - 주문 성공률, P95/P99 응답시간, 에러율, 최대 처리량을 핵심 KPI로 사용할 수 있다.
   - 재고 불일치 건수, 결제 실패율, 채팅 장애 중 주문 성공률처럼 비즈니스 흐름과 연결된 지표를 만들 수 있다.
   - Before/After 비교를 통해 HPA, Rate Limiting, Canary, rollback의 효과를 수치로 보여줄 수 있다.

3. 장애 시나리오가 자연스럽다.

   - 결제 승인 지연이나 실패는 외부 의존 서비스 장애를 가정하기 좋다.
   - 채팅 폭주나 장애는 핵심 구매 흐름과 부가 기능을 분리해 검증하기 좋다.
   - 알림 지연이나 실패가 발생해도 주문 완료 상태가 유지되는지 확인할 수 있다.

4. 팀 역할을 기능 구현보다 검증 중심으로 나눌 수 있다.

   - 프론트엔드, 백엔드, 인프라로 역할을 고정하지 않아도 된다.
   - 핵심 구매 흐름, 재고 정합성, Kubernetes 배포, 확장성, 관측성, 테스트 증거처럼 시나리오 중심으로 일을 나눌 수 있다.
   - 발표 산출물도 코드 양이 아니라 운영 증거와 검증 결과 중심으로 정리할 수 있다.

## 부트캠프 평가 관점

서비스 주제는 라이브 커머스지만 평가 중심은 앱 기능이 아니라 클라우드 네이티브 운영 능력이다.

| 관점 | 발표에서 보여줄 내용 |
| --- | --- |
| Kubernetes | Deployment, Service, Ingress/Gateway, ConfigMap/Secret, Probe, HPA |
| Private Cloud | 제한된 리소스에서 Pod 배치, 스케일링, 장애 격리, 내부 네트워크 통제 |
| 모니터링 | Prometheus로 서비스, Pod, 노드 지표 수집 |
| 관측성 | Grafana 대시보드, 로그 추적, request/order 기반 상관관계 |
| 배포 | CI/CD, GitOps 또는 manifest 기반 배포, Canary, rollback |
| 안정성 | 부하 테스트, 장애 주입, MTTR, 알림 |
| 보안/정책 | NetworkPolicy, ServiceAccount, Secret, Trivy scan |

## 서비스 범위

### MVP 핵심 흐름

```text
방송 시작 -> 스트리밍 상태 공개 -> 한정 상품 공개 -> 사용자가 주문 -> 재고 차감 -> 결제 승인 -> 주문 완료 알림
```

MVP에서는 실제 영상 송출 인프라 구현을 범위에서 제외한다. 대신 `streaming-service`가 mock 스트리밍 상태와 시청 세션을 제공하고, `live-service`가 방송 상태와 상품 오픈 이벤트를 관리한다. 프론트엔드 또는 API에서는 라이브 방송 중인 상황을 시뮬레이션한다.

### 서비스 분해

| 서비스 | 역할 | 핵심 지표 |
| --- | --- | --- |
| `gateway-service` | 인증, 라우팅, Rate Limiting | 요청 수, 차단 수, 에러율 |
| `live-service` | 방송 상태, 이벤트 시작/종료, 상품 오픈 관리 | 방송 상태 전환 시간, 이벤트 요청 수 |
| `streaming-service` | mock 스트리밍 상태, 시청 세션, 재생 상태 조회 | 시청 세션 수, 상태 조회 latency, 에러율 |
| `product-service` | 상품 정보, 가격, 재고 조회 | 재고 조회 latency, 재고 불일치 건수 |
| `order-service` | 주문 생성, 주문 상태 관리, 재고 차감 요청 | 주문 성공률, P95/P99, 주문 에러율 |
| `payment-service` | 결제 승인/실패 mock 처리 | 결제 성공률, 결제 실패율, 결제 지연 |
| `chat-service` | 채팅 메시지 수집과 조회 | 메시지 처리량, 채팅 latency, 에러율 |
| `notification-service` | 주문 완료, 결제 결과, 품절 알림 | 알림 성공률, 재시도 큐 적재량 |

## 앱과 운영 검증의 비중

이 프로젝트에서 앱은 클라우드 네이티브 검증을 위한 대상 시스템이다. 앱 기능은 얇게 두고, Kubernetes 운영 검증은 두껍게 둔다.

| 구분 | 범위 |
| --- | --- |
| 앱 기능 | 방송 상태, mock 스트리밍 상태, 상품 오픈, 상품 조회, 주문 생성, mock 결제, 단순 채팅, 알림 |
| Kubernetes | Deployment, Service, Ingress/Gateway, Probe, HPA, rolling update |
| 관측성 | Prometheus, Grafana, 로그 추적, Alertmanager |
| 배포 안정성 | CI/CD, GitOps 또는 manifest 기반 배포, Canary, rollback |
| 장애 대응 | 결제 지연, 채팅 장애, 주문 피크, 재고 경쟁, MTTR 측정 |

## 범위에서 제외

- 실제 영상 송출, WebRTC/HLS 인프라, 미디어 서버 구현
- 실제 PG 결제 연동
- 장기 운영 수준의 정산, 환불, 배송 관리
- 전체 커머스 백오피스 기능
- 운영 계약 수준의 장기 SLO 관리

## 주요 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 실제 영상 스트리밍 구현으로 범위가 커질 수 있음 | 영상 송출은 제외하고 mock 스트리밍 상태, 방송 상태, 이벤트 오픈 API로 시뮬레이션한다. |
| 결제 연동이 복잡해질 수 있음 | mock payment-service로 승인, 지연, 실패만 재현한다. |
| 재고 정합성 구현이 과도하게 복잡해질 수 있음 | DB lock, optimistic locking, Redis lock 중 하나만 선택한다. |
| 모든 PRD 요소를 다 넣으려다 발표가 흐려질 수 있음 | P0 시나리오를 주문 피크, 재고 정합성, 결제 장애, HPA로 고정한다. |
| 정량 지표가 환경에 따라 흔들릴 수 있음 | 절대 수치보다 동일 시나리오 Before/After 비교를 핵심 증거로 둔다. |

## 역할과 작업

워터폴 방식의 `프론트엔드 / 백엔드 / 인프라` 역할 구분은 이번 프로젝트에 맞지 않는다. 프로젝트의 핵심은 앱 기능을 순서대로 완성하는 것이 아니라, 라이브 이벤트 상황에서 운영 시나리오를 검증하는 것이다.

3명 팀이지만 역할을 3개로 고정하지 않는다. 대신 6개 작업 범위를 두고 각 범위마다 `주 담당`과 `부 담당`을 둔다.

| 작업 범위 | 주요 책임 | 핵심 산출물 |
| --- | --- | --- |
| 핵심 구매 흐름 | live, product, order, payment mock의 정상 E2E 흐름 | 주문 API, 상품 오픈 API, mock 결제, E2E 결과 |
| 재고 정합성과 장애 상태 | 재고 차감, 중복 주문 방지, 결제 지연/실패 상태 처리 | 초과 판매 0건 증거, `payment_pending` 흐름 |
| Kubernetes 배포 기반 | Deployment, Service, Ingress/Gateway, ConfigMap/Secret, Probe | 배포 manifest, readiness/liveness, 배포 smoke |
| 확장성과 배포 안정성 | HPA, rolling update, Canary, rollback, GitOps/ArgoCD | HPA Before/After, Canary 지표, rollback 기록 |
| 관측성과 알림 | Prometheus, Grafana, 로그 조회, Alertmanager | 대시보드, 알림 기록, request/order 추적 증거 |
| 테스트와 발표 증거 | k6, 장애 주입, 결과 캡처, 발표 KPI 정리 | k6 리포트, 장애 전후 비교표, 발표용 증거 인덱스 |

## 발표에서 가장 강한 조합

발표 시간이 제한된다면 다음 4개를 우선한다.

| 조합 | 이유 |
| --- | --- |
| 상품 오픈 피크 테스트 | 라이브 커머스에서 부하 테스트가 왜 필요한지 가장 직관적이다. |
| 재고 정합성 검증 | 동시성 문제가 비즈니스 손실과 직접 연결된다. |
| 결제 장애 격리 | 외부 의존 서비스 장애를 주문 유실 없이 통제하는 장면을 보여준다. |
| HPA Before/After | Kubernetes 기반 확장 효과를 수치로 보여준다. |

## 발표 구성안

1. 문제 제기: 라이브 상품 오픈 순간 주문과 채팅이 동시에 폭증한다.
2. 기준선 측정: 초기 부하 테스트에서 P99, 에러율, 주문 성공률을 측정한다.
3. 개선 적용: HPA, timeout, fallback, Rate Limiting, 관측성 대시보드를 적용한다.
4. 재검증: 동일 k6 시나리오로 Before/After KPI를 비교한다.
5. 장애 시연: 결제 장애 또는 채팅 장애를 주입하고 주문 흐름이 유지되는지 확인한다.
6. 운영 증거: Grafana, Alertmanager, 로그 추적으로 장애 감지와 원인 판단을 보여준다.
7. 배포 안정성: Canary 또는 rollback 시나리오로 변경 관리 안정성을 보여준다.
