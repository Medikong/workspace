# KT CloudNative Study 심화 프로젝트 실행 계획서

## 문서 목적

이 문서는 `00-PRD.md`의 요구사항을 5주 일정의 애자일 프로젝트로 실행하기 위한 상위 계획서다.

- `00-PRD.md`: 왜 하는가, 무엇을 만들어야 하는가
- `01-PRD_TRACEABILITY.md`: PRD의 필수/선택 항목이 어떤 Scenario/Epic/workplan으로 반영되는가
- `02-PROJECT_PLAN.md`: 어떤 목표를 어떤 순서와 검증 흐름으로 달성할 것인가
- Sprint/Milestone: 언제까지 어떤 덩어리를 끝낼 것인가
- Issue/Task: 누가 어떤 작업을 실제로 처리할 것인가

따라서 이 문서는 실제 이슈 목록이 아니라, 이슈를 만들기 전 기준점으로 사용한다.

## 프로젝트 한 줄 정의

기본 프로젝트에서 만든 MSA와 배포 기반을 AWS/GitOps 운영 흐름으로 승계하고, 장애·배포·보안·관측성 시나리오로 심화 프로젝트 목표를 검증한다.

## North Star

의료 정보 플랫폼 MSA가 장애, 배포, 보안, 트래픽 변화 상황에서도 관측 가능하고 통제 가능함을 증명한다.

## PRD 커버리지 원칙

이번 계획은 심화 프로젝트만 따로 떼어 보지 않고, 기본 프로젝트에서 심화 프로젝트로 이어지는 전체 흐름을 기준으로 산정한다.

- 기본 프로젝트의 대부분은 완료된 상태로 보되, 발표 가능한 증거가 있는지 Sprint 1 기준선에서 다시 확인한다.
- 기본 프로젝트에서 아직 완료되지 않은 LGTM/관측성은 심화 프로젝트의 핵심 잔여 범위로 끌어올린다.
- GitOps와 ArgoCD는 PRD 문장에 직접 드러나지 않더라도 AWS 시연 배포와 독립 배포 검증의 실행 경로로 다룬다.
- PRD의 필수/선택 항목 커버리지는 `01-PRD_TRACEABILITY.md`에서 추적한다.

## 시연 환경 원칙

PRD에는 명시되어 있지 않지만, 최종 시연 배포는 AWS에서 진행한다.

- 현재 상태는 Terraform 기반만 구성되어 있고, 아직 개발 서버와 QA 서버는 없다.
- 5주 일정에서는 완전한 dev/QA 분리보다 `AWS demo/QA 환경` 1개를 우선 구축한다.
- 로컬 또는 Docker/Kubernetes 환경은 빠른 검증 루프로 쓰되, 최종 발표 증거는 가능한 한 AWS 시연 환경에서 확보한다.
- AWS 환경 준비가 늦어지면 로컬 검증 결과를 백업으로 두되, 발표에서는 환경 차이와 남은 리스크를 명시한다.

## 문제 정의

기본 프로젝트는 MSA 구조를 구현하는 데 초점이 있었지만, 운영 관점에서는 다음 질문에 아직 충분히 답하지 못한다.

- Terraform 기반은 있으나 실제 개발/QA 서버가 없어 AWS 시연 환경을 먼저 세워야 한다.
- 기본 프로젝트의 완료분과 심화 프로젝트 신규 범위가 섞여 있어 요구사항 커버리지를 명시적으로 추적해야 한다.
- GitOps/ArgoCD 배포 경로가 계획에 드러나지 않으면 실제 시연과 운영 검증의 연결이 약해진다.
- LGTM/관측성은 기본 프로젝트에서 남은 작업이므로 심화 프로젝트의 핵심 증거로 완성해야 한다.
- 특정 서비스가 실패하면 영향 범위를 설명할 수 있는가?
- 한 서비스의 배포가 다른 서비스에 영향을 주지 않음을 검증할 수 있는가?
- 서비스 간 통신 경계와 권한 경계를 정책으로 강제할 수 있는가?
- 배포 전에 위험한 보안 설정을 차단할 수 있는가?
- 발표에서 "무엇을 만들었다"가 아니라 "무엇을 검증했다"를 보여줄 수 있는가?

## 목표

| 목표 | 설명 | 검증 방식 |
| --- | --- | --- |
| AWS 시연 환경 확보 | Terraform 기반을 검토하고 demo/QA 용도 AWS 실행 환경을 준비한다. | Terraform plan/apply 결과, 클러스터 접근, 배포 smoke |
| PRD 커버리지 확보 | 기본/심화 프로젝트 필수·선택 항목이 시나리오와 Epic으로 추적되게 한다. | PRD traceability matrix, Sprint Planning 갭 점검 |
| GitOps 배포 흐름 확보 | GitOps/ArgoCD로 manifest 변경, sync, rollback 흐름을 설명한다. | ArgoCD sync 상태, 배포 smoke, rollback runbook |
| 서비스 통신 통제 | Gateway, Service Mesh, NetworkPolicy로 호출 경로와 허용 범위를 관리한다. | 허용/차단 통신 시나리오 실행 |
| LGTM 관측 가능성 확보 | 메트릭, 로그, 대시보드, 가능하면 트레이스로 장애 지점을 식별한다. | 장애 주입 후 Grafana/Loki/Tempo 또는 대체 로그 확인 |
| 배포 안정성 확보 | 서비스별 독립 배포와 최소 가용성을 보장한다. | 단일 서비스 배포/E2E/PDB 검증 |
| 보안 기준 자동화 | CI에서 위험한 매니페스트와 코드 품질 문제를 조기에 차단한다. | Trivy/SonarQube 실패 케이스 검증 |
| 발표 가능성 확보 | 핵심 사용자 흐름과 운영 검증 결과를 하나의 이야기로 설명한다. | 발표 리허설과 결과 캡처 |

## MVP 범위

전체 일정은 5주지만 첫 주는 계획 기간이고 마지막 주는 최종 발표 준비 구간이다. 따라서 실험과 검증 결과는 2026-06-19(금)까지 확보하는 것을 기준으로, 각 영역에서 "운영 가능성을 증명하는 최소 단위"를 우선한다.

### Must Have

- AWS demo/QA 환경 구성 방향 결정 및 Terraform 기반 검증
- AWS 환경 클러스터 접근, 이미지 배포 경로, 기본 smoke 확인
- PRD 필수/선택 항목 커버리지 추적표 작성
- GitOps/ArgoCD 배포 경로, sync 기준, rollback 기준 정리
- Kubernetes 서비스 디스커버리와 Gateway/Ingress 라우팅 검증
- Service Mesh 도입 방향 결정 및 최소 트래픽 제어 검증
- Prometheus + Grafana 기반 서비스 메트릭 수집
- Loki 또는 대체 로그 중앙화 경로 확인
- 장애 주입 시 에러율/응답시간/영향 서비스 확인
- Trivy 기반 Kubernetes manifest 보안 스캔
- RBAC, ServiceAccount, NetworkPolicy 최소 권한 구성
- 서비스별 독립 배포 또는 독립 검증 흐름 정리
- 발표용 핵심 시나리오, 지표, 회고 자료 정리

### Should Have

- 분산 추적(Jaeger 또는 Tempo) 최소 도입
- Slack 알림 연동
- SonarQube 정적 분석 및 품질 게이트
- ArgoCD 애플리케이션 상태와 sync 이력 캡처
- Circuit Breaker 또는 Fallback 시나리오
- Newman 기반 E2E 자동 실행

### Could Have

- KEDA 기반 이벤트 소비자 오토스케일링
- ArgoCD Rollouts Canary 배포
- OPA Gatekeeper 정책 강제
- Falco 런타임 이상 행위 탐지
- OWASP ZAP DAST 보고서

### Won't Have

- 모든 서비스를 완전한 운영 수준으로 재작성
- 모든 선택 과업의 완성형 구현
- 실제 운영 환경 수준의 SLA/SLO 체계
- 장기 운영용 비용/용량 최적화

## 핵심 검증 시나리오

| 시나리오 | 질문 | 성공 기준 | 산출물 |
| --- | --- | --- | --- |
| AWS 시연 환경 | Terraform 기반만 있는 상태에서 AWS demo/QA 환경을 준비할 수 있는가? | AWS 환경 접근, 기본 배포, smoke 결과가 확인된다. | Terraform 결과, 클러스터 접근 로그, smoke 결과 |
| GitOps/ArgoCD 배포 | 서비스 변경을 GitOps 경로로 AWS 환경에 반영할 수 있는가? | ArgoCD sync 또는 대체 배포 경로가 검증되고 rollback 기준이 정리된다. | ArgoCD sync 캡처, manifest diff, rollback runbook |
| 정상 사용자 흐름 | 환자 조회, 예약, 알림 흐름이 정상 동작하는가? | E2E 요청이 성공하고 주요 서비스 로그가 확인된다. | E2E 결과, 요청 흐름 다이어그램 |
| 서비스 장애 | 처방/예약 서비스 장애 시 영향 범위를 설명할 수 있는가? | 실패 서비스, 에러율, 응답시간 변화가 대시보드에서 식별된다. | 장애 주입 결과, Grafana 캡처 |
| 통신 차단 | 허용되지 않은 서비스 간 호출을 막을 수 있는가? | NetworkPolicy 또는 Mesh 정책으로 비인가 통신이 실패한다. | 정책 manifest, 검증 로그 |
| 독립 배포 | 한 서비스 배포가 다른 서비스 흐름을 깨뜨리지 않는가? | 단일 서비스 재배포 후 핵심 E2E가 통과한다. | 배포 로그, 테스트 결과 |
| 보안 스캔 | 위험한 manifest가 배포 전에 차단되는가? | Trivy 실패 케이스가 CI 또는 로컬 검증에서 실패한다. | 스캔 결과, 실패 샘플 |
| 발표 리허설 | 기술 요소가 하나의 이야기로 설명되는가? | 문제, 목표, 시연, 지표, 회고 흐름이 10분 안에 연결된다. | 발표 스토리라인, 체크리스트 |

## 5주 일정 운영

### Phase 0: 계획과 백로그 확정

기간: 2026-05-27(수) ~ 2026-06-02(화)

목표: 심화 작업을 얹기 전에 MVP 범위, 검증 시나리오, Epic, workplan, 역할을 고정한다.

주요 작업:

- PRD 요구사항을 Must/Should/Could/Won't로 재분류
- PRD 요구사항 추적표 작성
- 핵심 사용자 흐름 1개와 장애 시나리오 1개 확정
- AWS demo/QA 환경 범위와 Terraform 검증 기준 확정
- GitOps/ArgoCD 배포 경로의 현재 상태와 gap 확인
- Epic 정의와 workplan YAML 작성
- 팀 역할, 소유 영역, 공통 파일 수정 규칙 정리
- 이슈 Ready 기준과 Definition of Done 확정

완료 기준:

- 팀이 같은 목표 문장과 MVP 범위에 합의한다.
- P0 시나리오와 후순위 과업이 분리된다.
- workplan의 `local_id`, `depends_on`, `done_when`, `evidence`가 준비된다.

### Sprint 1: 기준선과 E2E 확보

기간: 2026-06-03(수) ~ 2026-06-09(화)

목표: 정상 사용자 흐름을 먼저 재현해 장애, 보안, 배포 검증의 기준선을 만든다.

주요 작업:

- 서비스 배포 구조, Gateway/Ingress, 내부 DNS 흐름 확인
- AWS demo/QA 환경 접근과 기본 배포 smoke 확인
- GitOps/ArgoCD sync 경로 또는 단기 대체 배포 경로 확인
- 테스트 데이터 준비
- 기본 E2E 테스트 또는 수동 검증 절차 정리
- 정상 흐름 실행 결과 캡처
- 서비스 메트릭 수집 대상 인벤토리 작성

완료 기준:

- 핵심 사용자 흐름이 한 번 이상 재현된다.
- Sprint 2에서 구현할 이슈가 검증 시나리오 단위로 쪼개진다.

### Sprint 2: 운영 기능 구현과 정책 적용

기간: 2026-06-10(수) ~ 2026-06-16(화)

목표: 관측성, 통신 제어, 보안 스캔, 권한 정책을 실제 시스템에 적용한다.

주요 작업:

- Service Mesh 후보를 선택하고 최소 트래픽 제어를 검증
- AWS 시연 환경에서 P0 시나리오를 실행할 수 있는지 확인
- 서비스별 메트릭 수집과 Grafana 패널 구성
- Loki 로그 조회 경로와 Tempo/Tracing 도입 여부 결정
- 장애 주입 시나리오 실행 및 결과 캡처
- Trivy manifest scan 추가
- RBAC, ServiceAccount, NetworkPolicy 적용
- 독립 배포 또는 단일 서비스 재배포 검증
- 필요 시 Slack 알림, SonarQube, tracing 중 1~2개만 추가
- 2026-06-12(금) 중간 발표에서 초기 결과와 남은 리스크 공유

완료 기준:

- 장애, 보안, 통신 차단 중 최소 1개 이상은 실제 결과가 있다.
- 각 기능은 "설치 완료"가 아니라 "실패/성공 케이스 확인"으로 닫힌다.
- 2026-06-19(금)까지 나머지 P0 시나리오를 완료할 수 있는 상태로 쪼개져 있다.

### Sprint 3: 실험 결과 Freeze

기간: 2026-06-17(수) ~ 2026-06-19(금)

목표: 최종 발표에 사용할 모든 실험과 검증 결과를 고정한다.

주요 작업:

- 전체 E2E 흐름 재검증
- 장애/보안/통신 정책 시나리오 반복 실행
- Grafana 캡처, 로그, CI 결과, 정책 차단 결과 정리
- 독립 배포 검증 결과 또는 후속 과제 분류
- 최종 증거 인덱스 초안 작성

완료 기준:

- 최소 2개 이상의 운영 검증 결과가 최신 증거로 고정된다.
- P0 시나리오 중 미완성 항목은 후속 과제로 분류된다.
- 2026-06-22(월)부터 새 실험 없이 발표 준비로 전환할 수 있다.

### Final Prep: 발표 준비와 리허설

기간: 2026-06-22(월) ~ 2026-06-24(수)

목표: 새 기능과 새 실험을 추가하지 않고, 확보된 결과를 최종 발표 가능한 이야기와 자료로 정리한다.

주요 작업:

- 발표용 아키텍처 다이어그램과 핵심 요청 흐름 정리
- 검증 결과 캡처 목록 정리
- 회고 정리: 문제, 결정, 결과, 남은 과제
- 발표 리허설
- 발표 당일 시연 실패에 대비한 백업 캡처 준비

완료 기준:

- 최종 발표 흐름이 문제, 목표, 설계, 검증, 결과, 회고 순서로 연결된다.
- 발표 당일 사용할 캡처와 백업 자료가 준비되어 있다.
- 미완성 항목은 숨기지 않고 후속 과제로 설명된다.

## 애자일 운영 방식

### 스프린트 리듬

- Sprint Planning: 매주 시작 시 목표, 범위, 완료 기준 확정
- Daily Sync: 진행 상황, 막힌 점, 당일 검증 목표 공유
- Mid-Sprint Check: 시나리오가 실제로 재현되는지 중간 점검
- Sprint Review: 만든 기능보다 검증 결과 중심으로 공유
- Retrospective: 다음 주에 바꿀 행동 1개를 정한다

### 이슈 작성 원칙

이슈는 기능 이름이 아니라 검증 가능한 결과로 작성한다.

좋은 예:

- `prescription-service 장애 시 Grafana에서 에러율 증가를 확인한다`
- `NetworkPolicy로 patient-service -> notification-service 직접 호출을 차단한다`
- `Trivy config scan에서 privileged 컨테이너 manifest가 실패하게 만든다`

피해야 할 예:

- `Grafana 하기`
- `보안 하기`
- `Istio 설정`

### 이슈 템플릿

```text
목표:
검증할 질문:
작업 범위:
완료 기준:
의존 이슈:
발표에서 보여줄 방식:
```

## 역할 분리 기준

실제 담당자는 스프린트 계획 때 확정하되, 역할은 기능명이 아니라 책임 영역으로 나눈다.

| 역할 | 책임 |
| --- | --- |
| PM/프로젝트 리드 | 목표, 범위, 일정, 발표 스토리, 리스크 관리 |
| App/Service | 서비스 API, 장애 시나리오, 테스트 데이터, fallback 검증 |
| Platform/GitOps | Kubernetes manifest, Gateway, Mesh, 배포 구조 |
| Observability | Prometheus, Grafana, 로그, tracing, 알림 |
| Security/DevSecOps | Trivy, SonarQube, RBAC, ServiceAccount, NetworkPolicy |
| QA/Scenario | E2E 시나리오, 검증 체크리스트, 결과 캡처 |

한 사람이 여러 역할을 맡을 수 있지만, 한 이슈의 완료 기준과 발표 증거는 반드시 명확히 둔다.

## 의사결정 기준

일정에서 선택지가 충돌하면 다음 순서로 판단한다.

1. 발표에서 운영 능력을 증명하는 데 필요한가?
2. 핵심 검증 시나리오와 직접 연결되는가?
3. 팀이 1~2일 안에 재현 가능한가?
4. 실패했을 때 되돌릴 수 있는가?
5. 결과를 수치, 로그, 화면 캡처로 남길 수 있는가?

## 리스크와 대응

| 리스크 | 징후 | 대응 |
| --- | --- | --- |
| 기술 범위 과다 | Istio, tracing, OPA, Falco, KEDA를 모두 깊게 하려 한다. | Must Have 우선, Should/Could는 증거 확보 후 추가한다. |
| AWS 환경 지연 | Terraform 기반은 있으나 dev/QA 서버가 없어 시연 환경 준비가 늦어진다. | demo/QA 단일 환경을 우선 만들고, 로컬 검증 결과를 백업으로 유지한다. |
| 검증 없는 설치 | 도구 설치는 됐지만 시나리오 결과가 없다. | 모든 작업에 실패 케이스와 성공 케이스를 붙인다. |
| 통합 지연 | 각자 만든 기능이 마지막에 연결되지 않는다. | Sprint 1부터 핵심 E2E 흐름을 고정한다. |
| 발표 흐름 붕괴 | 기능 나열식 발표가 된다. | 문제, 목표, 시나리오, 결과, 회고 순서로 스토리를 관리한다. |
| 공통 파일 충돌 | 여러 명이 같은 manifest나 README를 수정한다. | 소유 영역과 공통 파일 수정 규칙을 스프린트 시작 때 정한다. |
| 실험 결과 지연 | 최종 발표 주간에 새 실험을 시도한다. | 2026-06-19(금)을 실험 결과 Freeze로 두고 이후에는 발표 준비만 한다. |

## 발표 스토리라인

발표는 기능 목록이 아니라 운영 능력 검증 흐름으로 구성한다.

1. 문제: 기본 MSA는 만들었지만 운영 가능성은 검증되지 않았다.
2. 목표: 장애, 배포, 보안, 관측성을 통제 가능한 상태로 만든다.
3. 설계: Gateway, Mesh, Observability, Security Policy의 역할을 분리한다.
4. 시연: 정상 흐름과 장애/차단/스캔 실패 시나리오를 보여준다.
5. 결과: 에러율, 응답시간, 로그, 정책 차단 결과를 근거로 제시한다.
6. 회고: 잘 된 점, 막힌 점, 다음 개선 과제를 정리한다.

## Definition of Done

이 프로젝트에서 완료는 코드 또는 manifest 작성만으로 판단하지 않는다.

- 요구사항 또는 시나리오와 연결되어 있다.
- 로컬 또는 클러스터에서 재현 가능한 검증 절차가 있다.
- 성공 케이스와 실패 케이스 중 최소 하나 이상이 확인됐다.
- 로그, 스크린샷, 테스트 결과, 대시보드 중 하나 이상의 증거가 남았다.
- 발표에서 어떻게 보여줄지 설명할 수 있다.
- 남은 리스크가 있으면 후속 과제로 명시했다.
