# KT CloudNative Study 5주 일정 마일스톤

## 문서 목적

이 문서는 `02-PROJECT_PLAN.md`를 실제 일정에 맞춰 계획 기간, 실행 스프린트, 발표 준비 구간으로 나누기 위한 마일스톤 계획서다.

- `02-PROJECT_PLAN.md`: 프로젝트 목표, 범위, 검증 전략
- `01-PRD_TRACEABILITY.md`: PRD 필수/선택 항목 충족 여부와 누락 항목 관리
- `03-MILESTONES.md`: 날짜별 마일스톤, 산출물, 통과 기준
- `05-scenarios/README.md`: 정상/장애/보안/배포 검증 시나리오
- `../workplans/*.yaml`: 이슈로 등록할 작업 의존관계

마일스톤은 개별 이슈 목록이 아니라, 각 구간이 끝났을 때 무엇이 검증되어야 하는지 정의한다.

## 운영 전제

- 전체 일정은 약 5주로 본다.
- 첫 주는 구현 스프린트가 아니라 계획 기간으로 분리한다.
- 실제 시연 배포는 AWS에서 진행한다.
- 현재 AWS는 Terraform 기반만 있고 개발/QA 서버는 없으므로, 데모 겸 QA 환경을 우선 목표로 둔다.
- 기본 프로젝트의 대부분은 완료된 상태로 보되, LGTM/관측성은 미완료 잔여 범위로 둔다.
- GitOps와 ArgoCD는 AWS 시연 배포, 독립 배포, 롤백 설명에서 핵심이 되는 절차로 포함한다.
- 마지막 주는 발표 준비 구간으로 보고 새 실험을 추가하지 않는다.
- 모든 실험과 검증 결과는 늦어도 2026-06-19(금)까지 준비한다.
- 중간 발표 이후에는 새 범위를 늘리기보다 검증 가능한 P0 시나리오 완성에 집중한다.

## Phase 0: 계획과 백로그 확정

### 목표

기술 구현에 들어가기 전에 프로젝트 범위, 검증 시나리오, Epic, workplan, 역할을 확정한다.

### 주요 질문

- 이번 심화 프로젝트가 증명하려는 운영 능력은 무엇인가?
- PRD 필수/선택 항목 중 현재 문서에 드러나지 않는 것은 무엇인가?
- 2026-06-19까지 준비할 실험 결과는 무엇인가?
- AWS 데모 겸 QA 환경은 어떤 최소 범위로 준비할 것인가?
- GitOps/ArgoCD 배포 경로는 현재 어디까지 되어 있고 무엇이 비어 있는가?
- 중간 발표에서 어떤 기본 상태와 초기 결과를 보여줄 것인가?
- 어떤 작업이 P0이고 어떤 작업이 후속 과제인가?

### 주요 산출물

- MVP 범위 합의안
- PRD 요구사항 추적표
- P0 검증 시나리오
- Epic 정의
- Epic별 workplan YAML
- AWS 데모 겸 QA 환경 범위와 Terraform 검증 기준
- GitOps/ArgoCD 배포 경로 누락 항목 목록
- 역할과 소유 영역
- 시작/완료 기준

### 통과 기준

- 팀이 프로젝트 한 줄 정의와 MVP 범위에 합의했다.
- P0 시나리오와 후순위 과업이 분리됐다.
- workplan의 `local_id`, `depends_on`, `done_when`, `evidence`가 준비됐다.

## Sprint 1: 기본 상태와 E2E 확인

### 목표

정상 사용자 흐름을 먼저 재현해 장애, 보안, 배포 검증의 출발점을 만든다.

### 주요 질문

- 서비스 DNS와 Gateway/Ingress 라우팅이 동작하는가?
- AWS 환경에 접근하고 기본 배포 상태를 확인할 수 있는가?
- GitOps/ArgoCD 동기화 경로 또는 단기 대체 배포 경로가 확인되는가?
- 정상 사용자 흐름을 반복 실행할 수 있는가?
- 관측성/정책/보안 검증에 쓸 기본 상태가 마련됐는가?

### 주요 산출물

- 서비스 DNS와 Gateway/Ingress 확인 결과
- AWS 환경 접근과 기본 배포 확인 결과
- ArgoCD 애플리케이션 동기화 상태 또는 대체 배포 경로 확인 결과
- 정상 사용자 흐름 E2E 절차
- 테스트 데이터
- 정상 흐름 실행 결과
- 메트릭 수집 대상 목록

### 통과 기준

- 정상 사용자 흐름이 한 번 이상 성공했다.
- 같은 절차를 다른 팀원이 따라 실행할 수 있다.
- Sprint 2에서 장애/보안/통신 정책 검증에 쓸 기본 상태가 있다.

## Sprint 2: 운영 기능과 정책 검증

### 목표

관측성, 통신 제어, 보안 스캔을 실제 시스템에 적용하고 중간 발표에서 초기 결과와 위험 요소를 공유한다.

### 주요 질문

- 장애가 발생했을 때 영향 범위를 관측할 수 있는가?
- LGTM 중 메트릭, 로그, 트레이스의 최소 관측 범위를 어디까지 준비할 수 있는가?
- 허용되지 않은 서비스 간 통신을 정책으로 차단할 수 있는가?
- 위험한 Kubernetes 매니페스트를 배포 전에 잡아낼 수 있는가?
- 중간 발표에서 방향과 초기 결과를 설명할 수 있는가?
- AWS 환경에서 P0 시나리오를 실행할 수 있는가?

### 주요 산출물

- Prometheus 수집 결과
- Grafana 대시보드 1차 완성본
- Loki 로그 조회 경로 또는 로그를 한곳에서 조회하기 위한 대체 기준
- Tempo/Jaeger 도입 여부 결정 기록
- 장애 주입 절차
- NetworkPolicy 또는 Mesh 책임 경계 결정
- 허용/차단 통신 정책
- Trivy 스캔 기준과 실패 샘플
- 중간 발표용 진행 요약
- AWS 환경 위험 요소와 백업 검증 계획

### 통과 기준

- 장애, 보안 스캔, 통신 차단 중 최소 1개 이상은 실제 결과가 있다.
- 나머지 P0 시나리오는 2026-06-19까지 완료할 수 있을 만큼 잘게 나뉘어 있다.
- 중간 발표에서 완료/진행/위험 요소/다음 조치를 구분해 설명할 수 있다.

## Sprint 3: 실험 결과 고정

### 목표

최종 발표에 사용할 모든 실험과 검증 결과를 2026-06-19(금)까지 고정한다.

### 주요 질문

- 장애/보안/통신 정책 결과가 최신 캡처와 로그로 남아 있는가?
- 독립 배포 검증을 핵심 또는 부록으로 넣을 수 있는가?
- 발표 자료보다 실제 결과가 먼저 준비되어 있는가?

### 주요 산출물

- 장애 관측 최종 캡처
- 통신 정책 적용 전과 후의 결과
- 보안 스캔 정상/실패 결과
- 독립 배포 검증 결과 또는 후속 과제 분류
- 최종 근거 자료 색인 초안

### 통과 기준

- 최소 2개 이상의 운영 검증 결과가 최신 근거 자료로 확정됐다.
- P0 시나리오 중 미완성 항목은 후속 과제로 분류됐다.
- 2026-06-22부터는 새 실험 없이 발표 준비로 전환할 수 있다.

## Final Prep: 발표 준비와 리허설

### 목표

새 기능과 새 실험을 추가하지 않고, 준비된 결과를 최종 발표에서 설명할 흐름과 자료로 정리한다.

### 주요 질문

- 발표를 듣는 사람이 운영할 수 있는 수준이 어떻게 나아졌는지 이해할 수 있는가?
- 모든 주요 주장이 로그, 지표, 캡처, 테스트 결과 중 하나로 뒷받침되는가?
- 시연 실패에 대비한 백업 캡처가 있는가?

### 주요 산출물

- 발표 흐름
- 아키텍처 다이어그램
- 검증 결과 캡처 목록
- 10분 발표 리허설 피드백
- 회고와 후속 과제
- 백업 캡처 목록

### 통과 기준

- 최종 발표 흐름이 문제, 목표, 설계, 검증, 결과, 회고 순서로 연결된다.
- 발표 당일 사용할 캡처와 백업 자료가 준비되어 있다.
- 미완성 항목은 숨기지 않고 후속 과제로 설명된다.

## 마일스톤별 의사결정 게이트

### Gate 1: 범위 고정

2026-06-02(화)까지 필수 범위와 권장 범위를 확정한다.

- 필수 범위는 Sprint 1~3 구현 대상으로 유지한다.
- 권장 범위는 발표 증거 준비에 직접 도움이 되는 것만 남긴다.
- 선택 범위는 중간 발표 이후 여유가 없으면 제외한다.

### Gate 2: 중간 발표 리셋

2026-06-12(금) 중간 발표 이후에는 범위를 늘리지 않고 검증 결과를 모으는 데 집중한다.

- 시나리오가 없는 기능은 우선순위를 낮춘다.
- 설치만 된 기능은 완료로 보지 않는다.
- 실패 케이스를 만들 수 없는 기능은 발표 핵심에서 제외한다.
- GitOps/ArgoCD와 LGTM는 동기화 결과, 로그/지표 조회 결과, 장애 전후 캡처가 있어야 완료로 본다.

### Gate 3: 실험 결과 고정

2026-06-19(금) 이후에는 새 실험을 추가하지 않는다.

- 남은 시간은 재현성, 캡처, 발표 흐름, 회고 정리에 쓴다.
- 미완성 기능은 숨기지 않고 후속 과제로 분류한다.
- 발표 시연이 불안정하면 화면 캡처와 로그를 백업 자료로 준비한다.
